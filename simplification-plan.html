Daily Dispatch Quiz — System Simplification Plan

Prepared for review · Daily Dispatch Quiz · Baltimore

July 2026

Background

The Daily Dispatch Quiz has grown from a simple quiz app into a multi-faceted publishing system handling question generation, email delivery, subscriber management, scoring, referrals, community features, and archiving. While all of this functionality is working, the cumulative complexity creates fragility: more moving parts means more ways for things to go wrong, and the admin panel has become difficult to navigate.

This document lays out a structured plan to simplify the system without losing any important functionality. Each item includes its costs and benefits so decisions can be made deliberately.

Part 1 — Sources of Complexity

Before proposing fixes, it helps to understand where the complexity actually lives.

1. One enormous file doing two jobs

news-quiz.html is approximately 9,000 lines of mixed HTML, CSS, and JavaScript. It simultaneously runs the public-facing quiz that players use and the private admin panel that the editor uses. These are completely different audiences with completely different needs, but they share a single file. Every player who loads the quiz page also downloads all the admin code.

2. The data layer reads and writes everything at once

The Postgres database uses a single table — essentially a key-value store — where large JSON blobs (subscribers, scores, quizzes, progress, drafts) are all read and written together. Almost every API request reads all of this data, modifies one part of it, then writes all of it back. This creates race conditions: if two players submit answers at the same moment, one write can silently overwrite the other.

3. Draft management spans two devices and two storage systems

The quiz draft lives in both the browser's localStorage (on whichever device the editor is using) and on the server. Keeping these in sync requires manual "Save Draft" and "Load Draft" operations, which have caused several bugs including held questions being lost and published questions reappearing after a refresh.

4. The email pipeline runs two parallel code paths

The A/B email test — Group A receives a standard email, Group B receives an email with the first question embedded — means every part of the email system (teaser generation, token creation, send logic, click tracking) is duplicated. This was the root cause of the July 1 outage, where a single malformed subscriber email address caused the entire subscriber batch to fail.

5. Subscribers and prospects are managed separately

Two separate lists with slightly different behavior, separate send loops, and separate admin UI sections — even though they receive nearly identical emails.

6. Score and progress are stored in multiple places

A player's score is calculated on the client side (for display), re-validated on the server, stored in the progress key, and mirrored to a separate scores key for the leaderboard. Keeping these consistent adds code and created the bug where scores showed as "10" for dates older than two days.

Part 2 — Simplification Plan

Items are ordered by impact-to-effort ratio. Items 1–4 can be executed without touching the data layer. Item 5 is the most significant change and warrants its own planning phase.

Item 1 — Reorganize the Admin Panel into tabs

What: Divide the admin panel into clearly labeled tabs — for example: Today's Quiz | Email | Players & Scores | Archive | Settings — so only one section is visible at a time. The long vertical scroll disappears.

Benefit: Immediately easier to navigate. You find things by section rather than by memory of where they appear in a long scroll. No logic changes whatsoever — purely a presentation improvement.

Cost: One to two days of HTML/CSS restructuring. Very low risk of breaking anything.

Verdict: Quickest win on the list. Good first step.

Item 2 — Split the admin panel into its own page

What: Move all admin code out of news-quiz.html into a separate admin.html served at /admin. Players load only the quiz page; the editor navigates directly to /admin. Shared utilities (date formatting, localStorage helpers) move to a small shared file.

Benefit: The player-facing quiz file shrinks by roughly half, making it faster to load and easier to work on. The admin page has one job. Security can be enforced at the route level rather than embedded in the quiz UI.

Cost: A few days of careful extraction. Requires identifying which functions are shared vs. admin-only vs. player-only. Low risk to the player-facing experience since those code paths are untouched.

Verdict: High impact on maintainability. Natural to do alongside Item 1 or shortly after.

Item 3 — Eliminate A/B email testing

What: Standardize on one email format — almost certainly Group B (the embedded first question version, since it is the more engaging format) — and remove all the A/B split logic, token generation for Group A, group assignment, and parallel send loops.

Benefit: The email sending pipeline currently runs two parallel code paths with token generation, validation, and tracking for each group. Removing one path cuts email code complexity by roughly 40%. The email system is where the July 1 outage happened — simpler email code means fewer failure modes.

Cost: You lose the ability to compare Group A vs. Group B performance going forward. However, the experiment has been running long enough to have drawn conclusions. The Group B token tracking (for click-through measurement) is worth keeping even if A/B testing goes away. Approximately one day of work.

Verdict: Good return on investment. Email reliability is the highest-stakes part of the system.

Item 4 — Make the draft server-first

What: Treat the server as the single source of truth for the draft at all times. The admin always reads from and writes to the server automatically. The browser's localStorage becomes a display cache only — not a synchronization mechanism. The manual "Save Draft" / "Load Draft" distinction disappears.

Benefit: Eliminates an entire class of cross-device synchronization bugs. Held questions no longer get lost when regenerating. Published questions no longer reappear after loading the draft on a different device. The code that manages draft state simplifies significantly.

Cost: Admin panel draft operations require a network connection, but this is already true for question generation and publishing, so it is not a real regression. Two to three days of refactoring the draft read/write calls in the admin panel.

Verdict: Fixes ongoing real pain. Medium effort with a clean payoff.

Item 5 — Replace the blob key-value store with proper Postgres tables

What: Give each major data type its own Postgres table with typed columns — separate tables for subscribers, scores, quizzes, progress, drafts — rather than storing everything as JSON blobs in a single store table that is read and written in batch.

Benefit: This addresses the root cause of several bugs. Proper tables allow atomic row-level updates (eliminating race conditions when multiple players submit answers simultaneously), fast targeted queries (instead of loading all data for every request), and natural data management (adding, removing, and updating individual records without loading the full dataset). Long-term, this makes the system substantially more reliable and scalable.

Cost: This is the most significant change — approximately one week of careful migration work touching nearly every route in server.js. Requires a data migration script, a staging test before going live, and a rollback plan. The risk is real but manageable with proper preparation.

Verdict: Highest long-term value of any item. Worth treating as its own planning phase rather than folding it into the other items.

Item 6 — Unify subscribers and prospects into one list

What: Replace the two separate email lists (subscribers and prospects) with a single list where each record has a status field: subscribed or prospect. The email sending loop becomes one loop with a conditional rather than two parallel loops.

Benefit: Simpler email code, one place to manage all email addresses in the admin panel, and a cleaner foundation for future email features.

Cost: Small data migration to tag existing records. About one day of refactoring. Naturally pairs with Item 5 since both involve restructuring data.

Verdict: Small but clean simplification. Best done alongside Item 5.

Item 7 — Simplify or remove the nudge email

What: The mid-day nudge email (sent to players who have not yet played today's quiz) has its own scheduling, tracking, and send logic that partly duplicates the morning email infrastructure. Options: remove it entirely, or reduce it to the simplest possible form.

Benefit: Removes a parallel email code path and reduces one more way the batch-send failure mode can recur.

Cost: Nudge emails drive real re-engagement plays. The cost depends on the data — if a meaningful percentage of daily plays come from the nudge, removing it will reduce player counts. Worth checking the analytics before deciding.

Verdict: Optional. Low cost if you decide to cut it, but check the data first.

Suggested Sequencing

Phase A (weeks 1–2): Items 1, 2, and 3. These are independent of each other and of the data layer. Each can be done, tested, and deployed without affecting the others. Combined, they remove roughly half the current complexity from the files you work with most.

Phase B (weeks 3–4): Item 4. Once the admin panel is cleaner and the email system is simplified, refactoring the draft to be server-first becomes a well-scoped change against a more navigable codebase.

Phase C (separate planning session): Items 5 and 6. Restructuring the data layer is the highest-value and highest-effort item. It deserves its own planning conversation, a migration script, staging testing, and a rollback plan before going live.

Item 7: Decide after reviewing nudge email analytics. Can be done at any point.
